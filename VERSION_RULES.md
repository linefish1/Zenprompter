# 版本管理规则与说明

## 版本号格式

我们采用**语义化版本控制（Semantic Versioning 2.0.0）**标准，版本号格式为：`v主版本号.次版本号.修订号`

格式：`vX.Y.Z`

- **X**：主版本号（Major） - 当有不兼容的API变更时递增
- **Y**：次版本号（Minor） - 当有向下兼容的新功能添加时递增
- **Z**：修订号（Patch） - 当有向下兼容的问题修复时递增

## 版本号示例

- `v1.0.0` - 第一个正式版本
- `v1.1.0` - 添加了新功能，向下兼容
- `v1.1.1` - 修复了bug，向下兼容
- `v2.0.0` - 有重大变更，不向下兼容

## 发布类型

### 1. 正式版本（Stable Releases）
- 版本号格式：`vX.Y.Z`
- 示例：`v1.0.0`, `v1.2.3`
- 用于生产环境，经过充分测试

### 2. 预发布版本（Pre-releases）
- 版本号格式：`vX.Y.Z-alpha`, `vX.Y.Z-beta`, `vX.Y.Z-rc`
- 示例：`v1.0.0-alpha.1`, `v2.1.0-beta.2`, `v1.5.0-rc.1`
- 用于测试新功能，不建议用于生产环境

### 3. 开发版本（Development Builds）
- 版本号格式：`vX.Y.Z-dev` 或 `vX.Y.Z-nightly`
- 示例：`v1.1.0-dev`, `v2.0.0-nightly.20240528`
- 每日构建或持续集成构建

## 版本发布流程

### 1. 开发阶段
- 版本号：`vX.Y.Z-dev`
- 功能开发和bug修复
- 仅内部测试使用

### 2. Alpha测试
- 版本号：`vX.Y.Z-alpha.N`
- 内部功能测试
- 不稳定，可能有重大bug

### 3. Beta测试
- 版本号：`vX.Y.Z-beta.N`
- 公开测试，邀请用户参与
- 主要功能稳定，可能有小bug

### 4. Release Candidate（RC）
- 版本号：`vX.Y.Z-rc.N`
- 发布候选版本
- 接近正式版，主要修复严重bug

### 5. 正式发布
- 版本号：`vX.Y.Z`
- 稳定版本，面向所有用户
- 经过充分测试和验证

## 版本号递增规则

| 变更类型 | 版本号变化 | 示例 |
|----------|------------|------|
| 重大不兼容变更 | 主版本号 +1 | `v1.2.3` → `v2.0.0` |
| 新增向下兼容功能 | 次版本号 +1 | `v1.2.3` → `v1.3.0` |
| 向下兼容的bug修复 | 修订号 +1 | `v1.2.3` → `v1.2.4` |
| Alpha测试版本 | 添加 `-alpha.N` | `v1.2.3` → `v1.2.3-alpha.1` |
| Beta测试版本 | 添加 `-beta.N` | `v1.2.3` → `v1.2.3-beta.1` |
| RC版本 | 添加 `-rc.N` | `v1.2.3` → `v1.2.3-rc.1` |

## GitHub Releases标签规则

我们的GitHub Releases使用以下标签格式：

- **正式版本**：`vX.Y.Z`（例如：`v1.0.0`）
- **预发布版本**：`vX.Y.Z-{alpha|beta|rc}.N`（例如：`v1.0.0-beta.1`）
- **自动构建版本**：`v{N}`（例如：`v1`, `v2`, `v3`）

## 版本管理工具

我们使用以下工具进行版本管理：

1. **GitHub Releases**：用于发布和管理版本
2. **GitHub Actions**：自动构建和发布流程
3. **更新服务**：应用内自动检查更新

## 如何检查更新

应用内置了自动更新检查功能：

1. 点击控制面板中的"更新"按钮
2. 应用会自动检查GitHub Releases
3. 如果有新版本可用，按钮会变成绿色并显示版本号
4. 点击绿色按钮即可下载最新版本

## 更新频率

- **正式版本**：每2-4周发布一次
- **bug修复版本**：根据需要随时发布
- **自动构建版本**：每次推送到main分支时自动构建

## 版本兼容性

我们致力于保持向下兼容性：

- 同一主版本号内的更新通常向下兼容
- 主版本号更新可能包含不兼容的变更
- 所有变更都会在更新日志中明确说明

## 贡献者指南

如果您想为项目做出贡献：

1. 从`main`分支创建新分支
2. 使用`vX.Y.Z-dev`格式进行开发
3. 提交Pull Request到`main`分支
4. 版本号由维护者在合并时确定
